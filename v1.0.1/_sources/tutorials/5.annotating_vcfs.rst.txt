
Tutorial 5: Annotating VCFs with Signature Assignments
=======================================================

This tutorial walks through **per-mutation signature assignment** using a pre-trained
MuTopia topographic model. For each mutation in a VCF file, the model estimates the
posterior probability that the mutation was generated by each component (signature),
and reports the estimated per-component mutation counts for the whole sample.

We also cover the **whitelist** feature, which lets you apply a whole-genome model
to samples sequenced with a targeted exome or panel assay.

How it works
------------

Assignment happens in two phases:

1. **Refitting (E-step)** — MuTopia iteratively estimates the per-component mutation
   counts :math:`N_k` for the sample, taking the trained model's local (locus-level)
   predictions as prior information. This is similar to standard NMF refitting, but
   constrained by the topographic prior learned during training.

2. **Per-mutation posteriors** — Given :math:`N_k`, the probability that each
   individual mutation was generated by component *k* is computed as the posterior
   :math:`\phi_{ki} \propto N_k \cdot p(\text{mutation}_i \mid k)`. The log posteriors
   are written to the output VCF as ``INFO`` fields.

Prerequisites
-------------

The following tools must be on your ``PATH``:

- ``bcftools``
- ``bedtools``
- ``mutopia`` (the MuTopia package, installed into the active conda environment)

You will need:

- A trained model (``trained_model.pkl``, from Tutorial 3 or the pre-trained file below)
- A reference G-Tensor dataset (``Liver.nc``)
- A sample VCF or BCF file

Download tutorial data
----------------------

The tutorial data bundle contains a pre-trained liver cancer model, the reference
G-Tensor, and an example VCF:

.. code-block:: bash

   # Download the tutorial data archive (≈ 2 GB)
   wget -O tutorial_data.tar.gz https://zenodo.org/records/TODO/files/tutorial_data.tar.gz
   tar -xzf tutorial_data.tar.gz

Alternatively, use the pre-built model and dataset from Tutorial 3.


Step 1: Initialize the dataset (one-time setup)
------------------------------------------------

Before annotating individual samples, run ``topo-model setup`` to pre-compute
the model's locus-level predictions (local variables and normalizers) and bake
them into the dataset. This is a one-time step per model/dataset pair — the
resulting file can be reused across many VCF samples.

.. code-block:: bash

   topo-model setup \
       tutorial_data/trained_model.pkl \
       tutorial_data/Liver.nc \
       tutorial_data/Liver_setup.nc \
       -@ 4

``Liver_setup.nc`` is a copy of the original dataset with pre-computed model
state appended. Annotation runs significantly faster when using this file rather
than the raw dataset.


Step 2: Annotate a VCF
----------------------

Run ``mutopia sbs annotate-vcf`` to assign mutations to components. Pass the
trained model, the initialized dataset, and the sample VCF:

.. code-block:: bash

   mutopia sbs annotate-vcf \
       tutorial_data/trained_model.pkl \
       tutorial_data/Liver_setup.nc \
       tutorial_data/sample.vcf.gz \
       -m tutorial_data/mutation-rate.hg38.bedgraph.gz \
       -o sample_annotated.vcf

The ``-m`` flag supplies a background mutation rate file used to cluster adjacent
mutations (kataegis) before ingestion. If you omit it, pass ``--no-cluster`` to
skip clustering.

While running, progress messages are printed to stderr:

.. code-block:: text

   Loading model ...
   Dataset has 279694 regions, 279694 after coord subsetting.
   Ingesting mutations ...
   Ingested 42318 mutations.
   Built sparse sample, 42318 nonzero entries.
   Loading dataset ...
   Whitelist applied: 279694/279694 regions have nonzero exposure.
   Setting up dataset (this may take a while) ...
   Computing per-mutation posteriors ...
   Estimated component fractions:
     SBS1    0.3421
     SBS5    0.2876
     SBS40   0.1503
     SBS22   0.0984
     ...
   Annotating VCF ...

The estimated component fractions are also stored in the VCF header:

.. code-block:: text

   ##mutopia_Nk=<Description="Estimated per-component mutation counts",SBS1=14474.2,SBS5=12175.0,...>

Each variant record receives one ``INFO`` field per component:

.. code-block:: text

   chr1  925952  .  G  A  .  PASS  logp_SBS1=-0.412;logp_SBS5=-1.893;logp_SBS40=-3.114;...


Reading the output
------------------

The ``logp_<component>`` values are log posterior probabilities normalized over
all components, so they sum to 0 in log-space (i.e., the un-logged posteriors
sum to 1). A value close to 0 means the mutation is very likely assigned to that
component; a strongly negative value means it is unlikely.

To retrieve the most probable component for each mutation:

.. code-block:: bash

   bcftools query -f '%CHROM\t%POS\t%REF\t%ALT\t%INFO\n' sample_annotated.vcf \
     | python3 - << 'EOF'
   import sys, re

   for line in sys.stdin:
       chrom, pos, ref, alt, info = line.rstrip().split('\t')
       fields = dict(f.split('=') for f in info.split(';') if '=' in f)
       logps = {k: float(v) for k, v in fields.items() if k.startswith('logp_')}
       best = max(logps, key=logps.get).replace('logp_', '')
       print(chrom, pos, ref, alt, best)
   EOF


Using a whitelist for panel and exome data
------------------------------------------

When mutations come from targeted sequencing (exome capture, gene panel), the
model must know which genomic regions were actually assayed. Without this
information, the model assumes the entire genome was observed, which biases
signature assignments because the model expects many more background mutations
than were detected.

Provide a BED file of assayed regions with ``--white-list``:

.. code-block:: bash

   mutopia sbs annotate-vcf \
       tutorial_data/trained_model.pkl \
       tutorial_data/Liver_setup.nc \
       sample.vcf.gz \
       --white-list capture_targets.bed \
       -o sample_annotated.vcf

The whitelist adjusts two things:

1. **Exposure correction** — Per-region exposure terms are scaled by the fraction
   of each 10 kb genomic window covered by the whitelist intervals, so the model
   penalizes the absence of mutations in uncaptured territory.

2. **Mutation filtering** — Mutations outside the whitelisted regions are excluded
   from the analysis, discarding off-target artefacts.

.. note::

   The whitelist BED file should use the same chromosome naming convention as
   your VCF and reference genome (``chr1`` vs ``1``). Use the ``-chr`` flag to
   add or remove a prefix if needed.


Multi-sample VCFs
-----------------

If your VCF contains calls for multiple samples, specify which one to annotate:

.. code-block:: bash

   mutopia sbs annotate-vcf \
       model.pkl dataset_setup.nc cohort.vcf.gz \
       -name SAMPLE_ID \
       -o SAMPLE_ID_annotated.vcf


Additional options
------------------

.. code-block:: text

   -m, --mutation-rate-file FILE   Background mutation rate bedgraph (for clustering)
   --no-cluster                    Skip kataegis mutation clustering
   -name, --sample-name NAME       Sample name in multi-sample VCF
   -chr, --chr-prefix PREFIX       Chromosome prefix to add (e.g. "chr")
   -fa, --fasta FILE               Override reference genome FASTA
   -@, --threads N                 Parallel threads (default: 1)
   --no-pass-only                  Include non-PASS variants
   --skip-sort                     Skip VCF sorting (VCF must be lexicographically sorted)
   -o, --output FILE               Output VCF path (default: stdout)

Run ``mutopia sbs annotate-vcf --help`` for the full option reference.
